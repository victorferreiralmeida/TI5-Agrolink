# 4. Modelagem e Projeto Arquitetural

A solução proposta para o sistema AGROLINK segue uma arquitetura cliente-servidor, composta por aplicações cliente que se comunicam com um backend central responsável pelo processamento das regras de negócio e pela persistência dos dados. O sistema possui duas interfaces principais de acesso: uma aplicação web desenvolvida em React, utilizada por produtores e gerentes para acompanhar e gerenciar as atividades da fazenda, e um aplicativo mobile desenvolvido em Flutter, destinado ao uso em campo para registro e consulta de informações operacionais.

Ambas as aplicações cliente se comunicam com o backend por meio de requisições HTTP, utilizando uma abordagem baseada em REST com polling para atualização de informações e notificações. Dessa forma, as operações tradicionais do sistema, como autenticação de usuários, cadastro e consulta de ocorrências, gerenciamento das informações da equipe e da fazenda, além do acesso ao chat e às notificações, são realizadas por meio dessa comunicação entre cliente e servidor.

O backend da aplicação é implementado com o framework Spring Boot e organizado como um monólito modular, ou seja, uma única API centralizada com separação interna por módulos de domínio. Entre os principais módulos do sistema estão autenticação (Auth), ocorrências, equipe, fazenda, chat e notificações. Essa organização permite manter a separação lógica das responsabilidades do sistema sem a necessidade de uma arquitetura baseada em microsserviços.

Para o armazenamento das informações, o sistema utiliza um banco de dados único, podendo operar com MySQL como principal solução de persistência e com H2 em ambiente de desenvolvimento local. Nesse banco são mantidos os registros de usuários, ocorrências, dados da equipe, informações da fazenda, mensagens e demais dados operacionais da plataforma.

Além disso, o sistema utiliza RabbitMQ exclusivamente no módulo de chat, atuando como broker de mensagens para apoiar a troca de mensagens entre os usuários. Já as funcionalidades de mapa são tratadas no lado do cliente, com o uso de Leaflet em conjunto com OpenStreetMap, permitindo a renderização e visualização das informações geográficas diretamente nas aplicações cliente, sem dependência de integração com a API do Google Maps no backend.

### Diagrama de componentes

O diagrama de componentes foi utilizado para representar a arquitetura da aplicação em nível macro, evidenciando os principais módulos de software, as interfaces expostas e os mecanismos de integração entre clientes, backend, banco de dados e mensageria. Esse diagrama foi escolhido porque permite demonstrar claramente a decomposição arquitetural da solução, a separação de responsabilidades e a forma como os componentes colaboram entre si.

Conforme apresentado na Figura 3, a solução foi estruturada como um **monólito modular Spring Boot**, organizada em camadas lógicas de **Cliente**, **API REST**, **Persistência** e **Mapas no cliente**. No lado do cliente, a aplicação disponibiliza dois pontos de acesso: uma **Web App**, desenvolvida em **React**, e uma **Mobile App**, desenvolvida em **Flutter**. Ambos se comunicam com o backend por meio de **REST/HTTPS**, com **polling** para atualização de chat e notificações.

O backend concentra-se em uma **única API REST** (Spring Boot), internamente organizada por módulos de domínio: **Autenticação**, **Ocorrências**, **Equipe**, **Fazenda**, **Chat** e **Notificações**. Cada módulo expõe endpoints REST e encapsula regras de negócio, serviços e repositórios JPA, mantendo separação lógica sem deploy independente de microsserviços.

Na camada de persistência, todos os módulos compartilham um **banco de dados único** (MySQL em produção/homologação ou H2 em desenvolvimento local), acessado via JDBC/Hibernate. Ocorrências, equipe, fazenda, mensagens e notificações residem no mesmo schema, com isolamento por `fazendaVinculoId` no nível de aplicação.

O módulo de **Chat** publica eventos na fila **RabbitMQ** (`agrolink.chat.messages`) após persistir a mensagem. O consumer roda no próprio backend; as interfaces web e mobile **não consomem a fila diretamente** — recebem as mensagens via polling da API. **Ocorrências** e **notificações** não passam pelo broker; são gravadas e consultadas diretamente pelo serviço e pelo banco.

As funcionalidades de mapa são renderizadas no **cliente** com **Leaflet** e tiles **OpenStreetMap/Esri**, sem integração com Google Maps API no backend. Coordenadas e polígonos (fazenda/setores) são persistidos como campos da API de ocorrências e fazenda.

## Estilos e padrões arquiteturais utilizados

Os principais estilos e padrões arquiteturais identificados no diagrama são:

- **Monólito modular**: uma aplicação Spring Boot com módulos de domínio bem delimitados (controllers → services → repositórios JPA).
- **API REST única**: clientes web e mobile consomem os mesmos endpoints HTTP/JSON.
- **Cliente-servidor**: React e Flutter como camada de apresentação; regras de negócio no backend.
- **Polling HTTP**: atualização periódica de chat e notificações nas interfaces (sem WebSocket no MVP).
- **Mensageria pontual**: RabbitMQ apenas no fluxo de publicação/consumo interno do chat.
- **Mapas no cliente**: Leaflet/OSM na camada de apresentação; backend armazena coordenadas e GeoJSON.

## Descrição dos artefatos do diagrama de componentes

Conforme o diagrama apresentado na Figura 3, as entidades participantes da solução são:

### Web App (React)

Componente cliente responsável pela interface web do sistema. Consome a API REST via proxy Vite (`/api` → `:8080`), apresenta dashboard, mapa, ocorrências, equipe, mensagens, relatórios e notificações. Renderiza mapas com Leaflet no navegador.

### Mobile App (Flutter)

Componente cliente para uso em campo. Comunica-se diretamente com a API REST (emulador/dispositivo), permitindo login, registro de ocorrências com geolocalização, mapa, chat e notificações.

### API REST · Spring Boot (monólito modular)

Componente central do backend. Expõe endpoints REST únicos e concentra autenticação, ocorrências, equipe, fazenda, chat e notificações em módulos internos (`AuthController`, `OcorrenciaController`, `EquipeController`, `FazendaController`, `ChatController`, `NotificacaoController`).

### Módulo de Autenticação

Responsável por login, cadastro, perfis de usuário (produtor, gerente, funcionário de campo) e validação de token. Senhas armazenadas com BCrypt.

### Módulo de Ocorrências

Núcleo funcional do sistema: criação, listagem, filtros, status (ABERTA/RESOLVIDA), prioridade, anexos e coordenadas geográficas vinculadas a setores da fazenda.

### Módulo de Equipe

Gestão de membros, convites e vínculo dos usuários à fazenda.

### Módulo de Fazenda

Cadastro de perímetro e setores (polígonos GeoJSON), base para mapa e registro de ocorrências.

### Módulo de Chat

Salas de conversa, envio de mensagens e histórico. Publica eventos no RabbitMQ após persistir; a UI atualiza por polling da API.

### Módulo de Notificações

Registro e consulta de notificações operacionais (novas ocorrências, convites, alterações). Persistência direta no banco; exibição no cliente por polling.

### Banco de Dados (MySQL · H2)

Repositório persistente único da aplicação. Armazena usuários, ocorrências, equipe, fazenda, mensagens de chat e notificações.

### RabbitMQ

Broker utilizado **somente** pelo módulo de chat (`agrolink.chat.messages`). Demonstra mensageria assíncrona no backend; não substitui a API para leitura das mensagens pelos clientes.

### Leaflet / OpenStreetMap (cliente)

Camada de mapas executada no navegador e no app mobile. Exibe ocorrências georreferenciadas e limites da fazenda sem dependência de Google Maps API.

Em síntese, o diagrama de componentes demonstra uma arquitetura **cliente-servidor** com **monólito modular** no backend: os clientes acessam uma API REST única, os domínios de negócio ficam separados em módulos internos, a persistência é centralizada em um banco e o RabbitMQ atua pontualmente no chat. Essa organização equilibra simplicidade de deploy (MVP acadêmico), clareza de responsabilidades e evolução futura por módulos.
