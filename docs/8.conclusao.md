## 8.1. Resumo executivo

O **AGROLINK** é um sistema inteligente de monitoramento e gestão rural desenvolvido no contexto acadêmico da disciplina de Projeto de Laboratório de Engenharia de Software (PLF). O projeto propôs substituir a comunicação informal e descentralizada — comum em fazendas de médio e grande porte — por uma plataforma estruturada que centraliza ocorrências operacionais, geolocalização, equipe e comunicação no campo.

Ao final do semestre, a equipe entregou um **MVP funcional** composto por:

- **Back-end** monolítico modular em Spring Boot (Java 21) com API REST;
- **Front-end web** em React 19 + Vite + TypeScript;
- **Aplicativo mobile** em Flutter para uso em campo;
- **Documentação acadêmica** completa (requisitos, modelagem, wireframes, solução, avaliação ATAM);
- **Artefatos de gestão** (TAP, EAP, atas semanais, sprints) e materiais de divulgação (README ilustrado, vídeo pitch).

O sistema atende ao objetivo central do projeto: oferecer uma visão única e rastreável das operações rurais, apoiando a tomada de decisão com dados concretos em vez de informações dispersas em canais informais.

## 8.4. Solução técnica

### 8.4.1. Arquitetura implementada

A solução adotou arquitetura **cliente-servidor** com API REST única consumida por dois clientes:

```
┌─────────────────┐     ┌─────────────────┐
│   Web (React)   │     │ Mobile (Flutter)│
│   Vite + TS     │     │   Dart          │
└────────┬────────┘     └────────┬────────┘
         │    HTTP/JSON (REST)   │
         └───────────┬───────────┘
                     ▼
         ┌───────────────────────┐
         │  Spring Boot 4.0.5    │
         │  (Monólito modular)   │
         │  Controller → Service │
         │  → Repository (JPA)   │
         └───────────┬───────────┘
                     │
         ┌───────────┴───────────┐
         │  H2 (dev) / MySQL     │
         │  RabbitMQ (chat)      │
         │  uploads/ (mídias)    │
         └───────────────────────┘
```

**Principais controllers:** `AuthController`, `OcorrenciaController`, `FazendaController`, `EquipeController`, `ChatController`, `NotificacaoController`, `UsuarioPerfilController`.

### 8.4.2. Stack tecnológica

| Camada | Tecnologias |
|--------|-------------|
| Back-end | Java 21, Spring Boot 4.0.5, Hibernate/JPA, BCrypt, Logback, RabbitMQ |
| Front-end web | React 19, TypeScript 5.9, Vite 7, Leaflet/react-leaflet |
| Mobile | Flutter (stable), SQLite (offline) |
| Banco de dados | H2 (desenvolvimento) / MySQL 8+ (produção) |
| Infraestrutura | Docker (RabbitMQ), ngrok (demo pública) |

### 8.4.3. Decisões arquiteturais relevantes

| Decisão | Justificativa | Tradeoff |
|---------|---------------|----------|
| Monólito Spring Boot em vez de microsserviços | Simplicidade e prazo acadêmico | Escalabilidade horizontal limitada |
| Leaflet/OSM em vez de Google Maps API | Custo zero e ausência de chave de API | Menos recursos que soluções comerciais |
| Polling HTTP em vez de WebSocket | Implementação mais rápida no MVP | Latência maior no chat |
| Token `agrolink-{id}` em vez de JWT | Integração rápida entre módulos | Segurança abaixo do padrão de produção |
| Multi-tenant lógico (`fazendaVinculoId`) | Infraestrutura simples | Dependência de filtros corretos em todos os endpoints |
| Offline-first nos clientes | Disponibilidade em áreas rurais | Complexidade de sincronização e consistência eventual |

## 8.6. Avaliação da arquitetura (ATAM)

A avaliação conduzida conforme o método **ATAM** ([`docs/7.avaliacao.md`](7.avaliacao.md)) analisou sete cenários de atributos de qualidade:

1. **Desempenho** — atendido em uso local; teste de carga formal pendente;
2. **Usabilidade** — web validada com avaliação heurística; mobile em paridade crescente;
3. **Interoperabilidade** — API REST única consumida por web e mobile com contrato estável;
4. **Manutenibilidade** — camadas bem definidas no back-end; lógica offline duplicada entre clientes;
5. **Segurança** — BCrypt e Bearer token; token MVP sem expiração/JWT;
6. **Isolamento multi-tenant** — filtro por `fazendaVinculoId` no nível de aplicação;
7. **Disponibilidade offline** — cache local + outbox + `clientUuid` para idempotência.

**Conclusão ATAM:** a arquitetura atende de forma **satisfatória** interoperabilidade, manutenibilidade estrutural e usabilidade web; atende de forma **parcial** desempenho sob carga, segurança avançada, multi-tenancy rigoroso e operação offline completa.

## 8.8. Dificuldades encontradas

Durante o desenvolvimento, a equipe enfrentou os seguintes desafios:

1. **Conectividade em campo:** áreas rurais com sinal instável exigiram implementação de modo offline (SQLite no mobile, localStorage na web) com fila de sincronização e idempotência via `clientUuid`.

2. **Paridade web/mobile:** manter duas stacks de UI (React + Flutter) aumentou o esforço para replicar fluxos equivalentes, especialmente no chat e nas notificações.

3. **Divergência arquitetura alvo vs. implementação:** o diagrama de componentes previa microsserviços, API Gateway, WebSocket e Google Maps; o MVP consolidou monólito, polling HTTP e Leaflet/OSM — tradeoffs conscientes de escopo e prazo.

4. **Autenticação simplificada:** o token `agrolink-{id}` acelerou a integração, mas limitou a robustez de segurança exigida em produção.

5. **Demonstração pública:** a dependência de ngrok e máquina local para demo exigiu checklist rigoroso (RabbitMQ, Spring Boot, Vite, túnel) antes de apresentações.

6. **Integração RabbitMQ:** configurar mensageria para o chat exigiu Docker e alinhamento entre consumer do back-end e polling dos clientes.

## 8.9. Lições aprendidas

1. **Centralizar regras de negócio no back-end** facilita evolução e garante paridade entre clientes — clientes devem ser consumidores finos da API.

2. **Documentação contínua** (atas, READMEs por módulo, rastreabilidade de requisitos) reduz retrabalho e sustenta avaliações acadêmicas como o ATAM.

3. **MVP bem delimitado** (sem ERP, sem IoT, sem IA) permitiu entregar valor real dentro do prazo, evitando escopo creep.

4. **Offline-first** é essencial para o domínio rural, mas introduz complexidade de consistência que deve ser planejada desde cedo.

5. **Avaliação heurística** antecipada na web orientou melhorias de usabilidade antes da entrega final.

6. **Trabalho em equipe distribuído** (6 integrantes) exigiu divisão clara por módulo (back, front, mobile, documentação) e comunicação semanal estruturada.

## 8.13. Considerações finais

O projeto **AGROLINK** cumpriu sua proposta central: transformar a comunicação operacional rural de um modelo reativo e informal para uma abordagem **estruturada, georreferenciada e rastreável**. O MVP entregue demonstra viabilidade técnica e valor de uso para produtores, gerentes e funcionários de campo, com funcionalidades essenciais operacionais em três plataformas integradas por uma API REST comum.

As limitações identificadas — token simplificado, ausência de microsserviços, operação offline parcial e falta de testes de carga — são **débitos arquiteturais conscientes**, compatíveis com o escopo e prazo de um projeto acadêmico, e estão documentados para orientar evoluções futuras.

A experiência consolidou competências em **engenharia de software**, **arquitetura cliente-servidor**, **desenvolvimento full-stack** (Java/Spring, React, Flutter), **gestão ágil de projetos** e **avaliação arquitetural (ATAM)** — contribuindo para a formação dos integrantes como profissionais capazes de conceber, implementar e avaliar sistemas de software no contexto do agronegócio digital.

Com o encerramento oficial registrado em **28/06/2026**, o repositório contém a **versão definitiva** do AGROLINK, pronta para entrega formal da disciplina e para servir como base para continuidade ou referência em projetos futuros.
